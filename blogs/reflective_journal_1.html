On My Project

Speaking of tooling, this brings me to my project. At the start, I filled out the proforma document, which was challenging in itself because it required detailed planning, which is very difficult when starting out. In hindsight, completing the proforma helped me focus on the project’s goals and provided a clear outline, so I am quite happy with the resulting document.

Just to put you in context: In a nutshell, my project aims to create a tool (in the form of a webpage) to help modellers search for and organize openEHR archetypes.

Basically, a modeller with a list of clinical data elements on hand should be able to find appropriate existing archetypes and plan out the use of existing archetypes, the need for change requests in archetypes, and the proposal of possible new archetypes.

The project is divided into two main parts:

  • Backend Automation:
    First, a program will be developed that periodically downloads archetypes from an established repository (in this case, the international CKM). Each archetype will be parsed, and relevant information extracted into a proprietary JSON dictionary. This process will be fully automated and invisible to the user.
  • Web Application:
    The second part is a web application that uses this dictionary to offer the following functionalities:
    • Fast search of archetypes and display of basic information
    • Visualization of dependencies (parent-child, exclusions, inclusions)
    • Recommendation of similar or related archetypes
    • Management of a collection of archetypes
    • Creation of “new archetypes” as placeholders, so users know which archetypes are missing and need to be created
    • Management of a checklist of predefined elements
    • Project-based organization of all activities

You can already see that this project will have many interconnected moving parts. I’m also realizing that it will touch on various technical aspects (parsing, automation, web hosting) which will certainly expand my programming skills.

The development is based on my experience with two of my earlier projects:

  • The AQL Manager (Live online here: https://martinkochdesign.github.io/openEHR_AQL_manager/; Explanation here: https://github.com/martinkochdesign/openEHR_AQL_manager)

  • Visualization of CKM contents (Example online here: https://martinkochdesign.github.io/CKM_content_visualization/)

Currently, I’m investing a lot of hours to reach a working prototype of both parts. The idea is to include as much user feedback as possible (if the feedback can be implemented in time is another story...).

I am always reminded of the aphorism of the “Ninety–ninety rule” (Tom Cargill, Bell Labs):
The first 90 percent of the code accounts for the first 90 percent of the development time. The remaining 10 percent of the code accounts for the other 90 percent of the development time.

This tongue-in-cheek observation leads me to speed development up in the beginning, because this might be beneficial to all the little details that a full-fledged GUI implicates in the end. However, this approach has an obvious drawback: it may create the impression that the project is “already done” early on, even though much work remains on the backend. If sometimes it looks like there are no changes in the tool, it is probable that I am working on some detail that is not perceived in the GUI.

As an interesting observation, I suspect that I am not only using the extracted archetype data for the tool, but I can also generate some valuable statistics, particularly regarding quality. Let me get the final numbers for the next journal or another publication, but I think I can pinpoint archetypes that might need minor changes to get a coherent format all over the International CKM repository.

Also, I’ve already noticed that about a third of archetypes lack keywords, which complicates semantic linking for the recommendation algorithm (don’t worry, I am working on a solution!).

So, what is planned as next steps? First of all, we have to lock in the proforma. I guess there are not going to be major changes, as we already discussed the scope of the project. Secondly, I have prepared a first questionnaire, which I am going to use to get feedback from stakeholders, so we can give more emphasis on aspects that are more important to future users. As already said, user feedback is important to this project.
