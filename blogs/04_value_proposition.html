Value Proposition

With the context of the proposed project in mind, a value proposition has to be created before going on with the project.

This proposition is informed by the Value Proposition Canvas from Strategyzer. The Value Proposition Canvas is a practical tool that helps systematically understand and address the needs of stakeholders. It works by mapping out two key areas: the "Customer Profile" (which identifies stakeholder jobs, pains, and gains) and the "Value Map" (which outlines how a solution’s products and services relieve pains and create gains). By identifying stakeholders’ pain points and desired gains, the canvas enables target proposed solutions at the clinical knowledge modelers' needs.


Value Proposition Canvas

The "Customer" side

The resulting canvas identified the following important points.

The Clinical Knowledge Modeler wants to create a data model structure, based on a list of needed data elements. For this the modeler has to:

  • Search and identify "official" existing archetypes that include fitting data elements
  • Identify possible necessary changes in existing archetypes to accomodate needed data elements
  • Identify the need for new archetypes that include data elements that cannot be found or included in existing archetypes

The most important gains for the modeler in determination of the data model structure are:

  • Re-usability: By re-using existing archetypes, the This leads also to a faster development.
  • Interoperability: The use of existing archetypes result in a higher interooperability between different systems.
  • Identify gaps: The model structure can reveal archetypes that have not been developed by now in the openEHR community.
  • Quality: Ultimately this approach results in a better quality in healthcare, by using interenationally developed models.

The most hindering factors for the modeler in this undertaking are:

  • No alternative search engines: The search for archetypes is contained to the CKM by now. The modeler is limited to few or no alternative search engines.
  • Search for related archetypes: Starting from an archetype, it is difficult to find related archetypes or to visualize the relation to other archetypes.
  • Fragmented workflow: The workflow of creating a data structure contains various programs (Excel, XMind, etc.) which are not specific for openEHR and information has to be treated manually.

The "Value" side

To add to the modelers' gains and to remedy and overcome pain points, special attention will given to:

  • Provide an easy to use interface for the webpage (search, planning and visualization tool).
  • No vendor lock-in for the input. The data analyzed can come from any ZIP file ource with ADL1.4 archetype files.
  • One project oriented software that can manage a custom collection of arcehtypes, including the assignment of elements to archetypes in the collection.
  • Visualization of the archetype dependencies (parent-child, inclusion, exclusion, etc.).
  • Fast search experience thanks to the lighweight archetype database.
  • Recommendation of related or similar archetypes, based on a semantic recommendation algorithm.
  • No vendor lock-in for the output. All output files (project save files, archetype data file, etc.) are going to be created in a readable format (most likely JSON).

The services that will be needed to respond to the modelers needs are the following:

  • Consultation of archetypes in the international CKM or any ZIP file source.
  • Parsing of ADL1.4 archetype files for the needed information.
  • Automatization of parsing for periodic update of archtype information.
  • Online hosting of the webpage for searching, planning and visualization of archetypes.

Hence the value proposition can be formulated as:

This project empowers clinical knowledge modelers to efficiently search, visualize, and plan with openEHR archetypes through an intuitive, vendor-neutral tool. By streamlining the workflow and enhancing discoverability, it accelerates model development, improves interoperability, and supports higher quality healthcare data models.
