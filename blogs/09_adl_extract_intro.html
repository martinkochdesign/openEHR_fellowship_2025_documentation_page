ADL extraction - Introduction

For the final search, visualization, and planning tool, data from an archetype must be extracted. Hence, there is a need for an application that can execute this extraction periodically.

For this purpose, the ADL extraction application must fulfill the following functions. In general, the application receives a collection of archetype files. The application extracts the files and parses them for the required information. Once all archetypes have been parsed, more information can be addedâ€”for example, by a recommendation algorithm that relates different archetypes to each other.

Once all information is accumulated, the application generates an archetype dataset file that can then be used by the visualization tool.


ADL extraction - General view

The functionalities can be specified further:

  • Data source: The data source for extraction must be vendor-neutral and should not depend on a particular platform. The most sensible approach is to allow extraction from any collection of archetype files in a ZIP file. This typically includes:
    • CKM: CKM allows exporting archetypes in bulk as a ZIP file.
    • GitHub: Archetype collections can be downloaded from a GitHub repository as a ZIP file. Most CKMs have an associated GitHub repository.
    • Local: A custom collection of archetypes that a user can select from a local terminal.
  • ADL parsing: The application must be capable of parsing ADL 1.4-compliant archetype files in bulk.
  • Data extraction: Data extraction from the parsed archetype files must be limited to elements defined further on. In this case, only information from the English version of the archetype will be extracted. If the archetype is not available in English, the original language will be used.
  • Recommendation: A basic recommendation should be provided to link similar or related archetypes. Linking will be based on semantics (for example, using the keywords of each archetype). The recommendation will be hard-coded into the resulting dataset. In the future, a more dynamic recommendation might be implemented, where the visualization tool adds recommendations based on user input.
  • GUI: The application needs a GUI so a user can create the dataset manually whenever needed.
  • Automation: In addition to manual execution, the application should have a headless mode that does not require user input. In this mode, the application should be executed automatically to update the dataset periodically. The execution does not have to be triggered by the application itself, but can be managed by external services.

The planned developement will take place predominantly in Python code. As a result, the python code will be provided as well as a compiled Windows compatible executable file.
